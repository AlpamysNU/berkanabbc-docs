    SRS — Software Requirements Specification¶

    Version: 1.0 Project: Berkana Brain Date: 2026-05-25

    Важно

    Berkana Brain — инструмент скрининга. Результаты не являются медицинским диагнозом.

    1. Introduction¶

    1.1 Purpose¶

    Berkana Brain — веб-платформа нейрофункционального скрининга развития детей. Родители проходят структурированные опросники; система автоматически анализирует ответы и формирует профиль развития ребёнка с рекомендациями. Специалисты (психологи, педагоги) просматривают и валидируют результаты.

    1.2 Scope¶

    Система охватывает: - Возрастные модули скрининга: 0-6, 7-12, 13-24, 25-36, 37-48 месяцев - 3 роли пользователей: родитель, специалист, администратор - 3 языка интерфейса: RU / EN / KZ

    1.3 Non-medical disclaimer¶

    Все экраны с результатами скрининга и сгенерированные PDF обязаны содержать явную пометку: «Результаты скрининга не являются медицинским диагнозом».

    2. Functional Requirements¶

    FR-01: Telegram Magic Code Authentication¶

    • Пользователь вводит номер телефона на сайте
    • Telegram Bot присылает 4-значный код (TTL 10 мин)
    • Код подтверждает личность; JWT выдаётся автоматически
    • Регистрация и вход — один flow, новый аккаунт создаётся при первом входе
    • Роли: parent (по умолчанию), specialist, admin

    FR-02: User Profile + Child Profile Management¶

    • Профиль пользователя: ФИО (отдельные поля), email, страна, город
    • Скрининг недоступен без заполненного профиля (profile_complete = True)
    • Профиль ребёнка: ФИО отдельными полями, дата рождения, прикреплённый врач (опционально)
    • Скрининг недоступен без созданного профиля ребёнка

    FR-03: Age-Based Screening Module Selection¶

    • При старте скрининга система вычисляет возраст ребёнка в месяцах
    • Автоматически выбирается модуль с age_min_months ≤ age ≤ age_max_months
    • Если подходящего модуля нет — возвращается 400 с описанием

    FR-04: Screening Session — Start / Resume / Complete¶

    • POST /api/screening/sessions/ — создаёт сессию или возобновляет in_progress
    • POST /api/screening/sessions/{id}/answers/ — upsert ответа (autosave)
    • POST /api/screening/sessions/{id}/complete/ — завершает сессию + запускает generate_report()
    • Resume: при повторном открытии показывается первый неотвеченный вопрос
    • Типы вопросов: berkana_4opt, yes_no, scale_1_5, multiple_choice, text

    FR-05: Report Generation with Zones¶

    • После завершения сессии автоматически генерируется Report
    • Домены: motor_gross, motor_fine, speech, social, self_care, cognitive, sensory
    • Зоны: 🟢 typical (≥75%), 🟡 monitor (40–74%), 🔴 follow_up (<40%)
    • Поля отчёта: domain_scores, overall_level, interpretations, recommendations, delta_scores
    • delta_scores — разница с предыдущим скринингом (↑↓→)

    FR-06: PDF Export with Watermark¶

    • GET /api/screening/reports/{id}/pdf/ — генерирует PDF через WeasyPrint
    • 3 страницы: краткий итог, детальный анализ по доменам, рекомендации
    • Водяной знак: «Research Use Only / BND»
    • Дисклеймер «не является диагнозом» на каждой странице

    FR-07: Specialist Cabinet¶

    • Специалист видит только назначенных ему клиентов (SpecialistAssignment)
    • Назначение происходит автоматически при завершении скрининга
    • Специалист может: подтвердить отчёт / скорректировать / оспорить (ValidationRecord)
    • До верификации администратором специалист видит баннер-предупреждение
    • Консультации: специалист создаёт ConsultationSlot, родитель выбирает время

    FR-08: Admin Cabinet¶

    • KPI Dashboard: счётчики пользователей, скринингов, отчётов
    • Верификация специалистов (статусы: pending / verified / rejected)
    • Фильтр родителей rescreen_due — кому пора повторить скрининг
    • Назначение специалистов вручную

    FR-09: Screening Module Import via Admin UI¶

    • Администратор загружает .docx шаблон через /admin/screening/import
    • Двухфазный импорт: preview (список вопросов, ошибки) → confirm (запись в БД)
    • Шаблон berkana_module_template_v2.docx скачивается прямо из UI
    • Идемпотентный update_or_create — повторный импорт обновляет существующий модуль

    FR-10: Feedback Survey¶

    • После завершения скрининга показывается опрос из 5 вопросов
    • Возможность пропустить
    • Ответы сохраняются в FeedbackSurvey + FeedbackAnswer

    3. Non-Functional Requirements¶

    NFR-01: Security¶

    • JWT (SimpleJWT) для всех защищённых endpoints
    • DRF rate limiting: анонимный 30 req/min, авторизованный 120 req/min, auth endpoints 5 req/min
    • В production (DEBUG=False): HTTPS redirect, HSTS (1 год), secure cookies, XSS filter
    • Загружаемые файлы: только PDF/JPG/PNG, максимум 10 МБ
    • Медиафайлы в production — только через Nginx, не Django

    NFR-02: Internationalisation¶

    • Весь UI доступен на RU / EN / KZ
    • Тексты вопросов и интерпретаций хранятся в трёх языковых полях (_ru, _en, _kz)
    • Fallback: если EN/KZ отсутствует — показывается RU
    • i18next на фронтенде, useTranslation обязателен на всех страницах

    NFR-03: Performance¶

    • Rate limiting предотвращает перегрузку
    • Кэш для Telegram-кодов (FileBasedCache, TTL 600 с в dev)
    • Import preview кэшируется на 30 мин (FileBasedCache)

    NFR-04: Non-medical disclaimer¶

    • Дисклеймер виден: на странице опросника, на странице результатов, в PDF
    • Amber-баннер на ReportDetail.tsx

    4. Constraints and Assumptions¶

    # Ограничение
    C-01 База данных — только PostgreSQL. SQLite не поддерживается для миграций и тестов
    C-02 Вход только через Telegram Bot — нет email/пароль авторизации
    C-03 Все тексты интерпретаций проверяются специалистом перед публикацией
    C-04 Персональные данные детей защищены на уровне Django permissions (IsOwnerOrAdmin)
    C-05 Celery/Redis (очередь уведомлений) — Фаза 2, в MVP не реализована
    C-06 Соответствие Приказу МЗ РК № 152 от 26.11.2025 по дистанционным мед. услугам — в разработке
