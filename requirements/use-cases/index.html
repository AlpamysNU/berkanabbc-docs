    Пользовательские сценарии (Use Cases)¶

    Источник: BRD.pdf, раздел 14.

    UC-01 — Регистрация / Вход пользователя¶

    Актор: Родитель / Специалист

    Описание: Пользователь получает доступ к платформе через Telegram Bot.

    Основной поток: 1. Пользователь открывает сайт → страница /login 2. Вводит номер телефона 3. Frontend вызывает POST /api/auth/request-phone/ 4. Сайт показывает Telegram deep link и пытается открыть бота 5. Пользователь нажимает /start в боте по deep link 6. Пользователь делится номером телефона через Telegram contact button 7. Бот присылает 4-значный код 8. Пользователь вводит код на сайте 9. Если новый пользователь или профиль не заполнен: → /dashboard/profile 10. Если существующий профиль заполнен: → /dashboard

    Альтернативные потоки: - Код истёк (10 мин) → кнопка "Отправить новый код" - Неверный код → сообщение об ошибке, до 5 попыток - Сессия входа потеряна → запросить новую Telegram-ссылку - Номер Telegram contact не совпадает с введённым на сайте → ввести тот же номер и запросить новую ссылку

    Результат: Создана учётная запись (новый) или восстановлен доступ (существующий).

    Реализация: auth-telegram.md

    UC-02 — Заполнение профиля пользователя¶

    Актор: Родитель

    Описание: После первого входа пользователь заполняет профиль — обязательно для доступа к скринингу.

    Основной поток: 1. Пользователь переходит на /profile 2. Заполняет: ФИО, телефон, страна, город 3. Выбирает предпочитаемый язык 4. Нажимает "Сохранить" 5. profile_complete = True автоматически 6. Появляется CTA "Добавить ребёнка"

    Результат: Профиль сохранён, доступ к скринингу разблокирован.

    Бизнес-правило: Правило 1

    UC-03 — Добавление профиля ребёнка¶

    Актор: Родитель

    Описание: Создание профиля ребёнка для проведения скрининга.

    Основной поток: 1. Пользователь → /children 2. Нажимает "Добавить ребёнка" 3. Вводит: ФИО ребёнка, дата рождения, пол 4. Сохраняет 5. Система вычисляет age_in_months автоматически

    Результат: Профиль ребёнка создан, появляется кнопка "Начать скрининг".

    Примечание: Можно добавить несколько детей.

    UC-04 — Прохождение скрининга¶

    Актор: Родитель

    Описание: Основной сценарий — прохождение скринингового опросника.

    Основной поток: 1. Пользователь → /screening 2. Выбирает ребёнка из списка 3. Система автоматически определяет модуль по возрасту (Правило 3) 4. Показывает информацию о модуле ("Ранний возраст, 25 вопросов") 5. Пользователь нажимает "Начать" 6. Система создаёт ScreeningSession (или возобновляет существующую) 7. Вопросы отображаются по одному 8. Пользователь отвечает → каждый ответ сохраняется мгновенно 9. Прогресс-бар показывает прогресс 10. После последнего вопроса → кнопка "Завершить" 11. Пользователь нажимает "Завершить" → сессия закрывается

    Альтернативный поток (Resume): - Пользователь закрыл браузер → вернулся → сессия восстанавливается с того места, где остановился

    Результат: Все ответы сохранены. Запускается UC-05.

    Бизнес-правила: Правило 2, 3

    UC-05 — Автоматический анализ результатов¶

    Актор: Система

    Описание: Система анализирует ответы и формирует профиль развития.

    Основной поток: 1. Срабатывает при POST /api/screening/sessions/{id}/complete/ 2. Система собирает все ответы сессии 3. Группирует по доменам (motor, cognitive, social, speech, sensory) 4. Вычисляет domain_score = (sum_scores / max_possible) * 100 5. Находит подходящий ScoringRule для каждого домена 6. Определяет overall_level = наихудший из доменов 7. Если есть предыдущая сессия → вычисляет delta_scores 8. Назначает next_rescreen_at (Правило 6)

    Результат: Сформированы данные для отчёта.

    UC-06 — Генерация отчёта¶

    Актор: Система

    Описание: Создание структурированного отчёта по результатам анализа.

    Основной поток: 1. После UC-05 система создаёт объект Report 2. Собирает: domain_scores, interpretations, recommendations 3. Добавляет delta_scores (если повторный скрининг) 4. Сохраняет в БД 5. Создаёт уведомление report_ready 6. Возвращает report_id фронтенду

    Результат: Отчёт доступен в личном кабинете.

    UC-07 — Просмотр отчёта¶

    Актор: Родитель

    Описание: Пользователь просматривает результаты скрининга.

    Основной поток: 1. Пользователь → /reports 2. Видит список всех отчётов (по всем детям, по датам) 3. Нажимает на отчёт → /reports/{id} 4. Видит: - Радар-чарт с 5 доменами - Цветовую индикацию (зелёный / жёлтый / красный) - Интерпретации по каждому домену - Список рекомендаций (по приоритету) - Динамику (если повторный скрининг) - Дисклеймер: "Не является медицинским диагнозом"

    Результат: Пользователь понимает профиль развития ребёнка.

    UC-08 — Получение уведомления о повторном скрининге¶

    Актор: Система + Родитель

    Описание: Система информирует о необходимости повторного скрининга.

    Основной поток: 1. При завершении скрининга система записывает next_rescreen_at. 2. Администратор видит родителей, которым пора повторить скрининг, через /admin/parents?screening=rescreen_due. 3. Менеджер прозванивает родителей и просит пройти повторный скрининг. 4. Автоматическое in-app/Telegram уведомление родителю остаётся планируемым расширением Фазы 2.

    Интервалы: определяются ScreeningModule.rescreen_interval_days (14 / 30 / 90 дней)

    Бизнес-правило: Правило 6, 7

    UC-09 — Повторный скрининг¶

    Актор: Родитель

    Описание: Прохождение повторного скрининга с формированием сравнительного отчёта.

    Основной поток: 1. Пользователь получает уведомление (UC-08) 2. Нажимает ссылку → /screening 3. Проходит новый скрининг (UC-04) 4. Система определяет предыдущую завершённую сессию для этого ребёнка+модуля 5. Вычисляет delta_scores для каждого домена 6. Формирует сравнительный отчёт (UC-06) 7. В отчёте отображается динамика (↑ улучшение / ↓ снижение / → без изменений)

    Результат: Пользователь видит динамику развития ребёнка.
