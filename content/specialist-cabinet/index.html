    Кабинет специалиста¶

    1. Роль и задачи специалиста¶

    Специалист (психолог / педиатр / специалист по развитию) — отдельная роль в системе (role='specialist').

    Задачи: 1. Верификация своего профиля (загрузка документов → ручное одобрение admin) 2. Просмотр отчётов назначенных клиентов (родителей) 3. Валидация анализа: подтвердить / опровергнуть / добавить клинический комментарий 4. Аналитика: агрегированные данные по клиентам (Phase 3)

    Специалист НЕ: - Не видит данных неназначенных родителей - Не может изменять вопросы или скоринг - Не ставит диагнозы (платформа экспериментальная)

    2. Дизайн (test.pen)¶

    Desktop — cnPOU (Berkana Brain - Specialist Profile - Desktop)¶

    • Левая навигация: Dashboard, Screening Reviews, Children Profiles, Messages, Profile, Logout
    • Профиль: Фото, имя, специализация, статус Verified / Pending
    • Лицензия, страна, опыт, рейтинг эксперта
    • Кнопки: "Full profile", "Upload documents", "Save changes"
    • Форма редактирования: First name, Last name, Email, Phone, Country, Specialization, License number, Years of experience, Bio, Languages, Photo upload
    • Документы: Professional license, Diploma, Certification (Uploaded / Pending / Rejected)
    • Verification progress bar: 72%

    Tablet — BzRFd¶

    • Profile Summary: имя, специализация, опыт, страна
    • Статус: Pending verification
    • Edit profile | Upload docs | Save buttons
    • Форма: First name, Last name

    3. Модели базы данных¶

    SpecialistProfile — профиль специалиста¶

    class SpecialistProfile(models.Model):
    STATUSES = [
        ('unverified', 'Не верифицирован'),
        ('pending', 'На проверке'),
        ('verified', 'Верифицирован'),
        ('rejected', 'Отклонён'),
    ]
    SPECIALIZATIONS = [
        ('pediatric', 'Детский специалист'),
        ('neurologist', 'Невролог'),
        ('speech_therapist', 'Логопед'),
        ('psychologist', 'Психолог'),
        ('educator', 'Педагог'),
        ('other', 'Другое'),
    ]

    user = models.OneToOneField(UserProfile, on_delete=models.CASCADE, related_name='specialist_profile')
    specialization = models.CharField(max_length=30, choices=SPECIALIZATIONS)
    license_number = models.CharField(max_length=100, blank=True)
    years_of_experience = models.IntegerField(default=0)
    bio = models.TextField(blank=True)
    languages = models.CharField(max_length=200, blank=True)  # "ru, kz, en"
    country = models.CharField(max_length=100, blank=True)
    photo = models.ImageField(upload_to='specialist_photos/', null=True, blank=True)

    # Верификационные документы
    license_doc = models.FileField(upload_to='specialist_docs/', null=True, blank=True)
    diploma_doc = models.FileField(upload_to='specialist_docs/', null=True, blank=True)
    cert_doc = models.FileField(upload_to='specialist_docs/', null=True, blank=True)

    # Статус
    verification_status = models.CharField(max_length=20, choices=STATUSES, default='unverified')
    verified_at = models.DateTimeField(null=True, blank=True)
    verified_by = models.ForeignKey(
        User, null=True, blank=True, on_delete=models.SET_NULL,
        related_name='verified_specialists'
    )

    # Рейтинг (вычисляется из ValidationRecord)
    expert_score = models.DecimalField(max_digits=3, decimal_places=1, default=0)

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        verbose_name = 'Профиль специалиста'
        verbose_name_plural = 'Профили специалистов'

    SpecialistAssignment — назначение специалиста к родителю¶

    class SpecialistAssignment(models.Model):
    specialist = models.ForeignKey(
        UserProfile, on_delete=models.CASCADE,
        related_name='assignments', limit_choices_to={'role': 'specialist'}
    )
    parent = models.ForeignKey(
        UserProfile, on_delete=models.CASCADE,
        related_name='specialist_assignments', limit_choices_to={'role': 'parent'}
    )
    assigned_by = models.ForeignKey(
        User, null=True, blank=True, on_delete=models.SET_NULL
    )  # admin назначает, или родитель сам выбирает
    assigned_at = models.DateTimeField(auto_now_add=True)
    is_active = models.BooleanField(default=True)

    class Meta:
        unique_together = [('specialist', 'parent')]
        verbose_name = 'Назначение специалиста'

    ValidationRecord — запись валидации отчёта¶

    class ValidationRecord(models.Model):
    DECISIONS = [
        ('confirmed', 'Подтверждено'),   # Специалист согласен с анализом
        ('modified', 'Скорректировано'), # Анализ верен, но требует поправок
        ('disputed', 'Оспорено'),        # Анализ не соответствует наблюдениям
    ]

    report = models.ForeignKey('screening.Report', on_delete=models.CASCADE, related_name='validations')
    specialist = models.ForeignKey(UserProfile, on_delete=models.CASCADE, related_name='validations')
    decision = models.CharField(max_length=20, choices=DECISIONS)

    # Комментарий специалиста (клинический)
    comment = models.TextField(blank=True)

    # Если специалист видит red flags которые алгоритм пропустил
    additional_flags = models.JSONField(null=True, blank=True)
    # {'motor': 'follow_up', 'speech': 'follow_up'} — скорректированные зоны

    validated_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        unique_together = [('report', 'specialist')]
        verbose_name = 'Запись валидации'
        verbose_name_plural = 'Записи валидации'

    4. API endpoints для специалиста¶

    # Профиль специалиста
GET  /api/specialist/profile/                  — получить свой профиль
PUT  /api/specialist/profile/                  — обновить профиль
POST /api/specialist/profile/upload-doc/       — загрузить документ (license/diploma/cert)

# Назначения (клиенты)
GET  /api/specialist/assignments/              — список назначенных родителей
GET  /api/specialist/assignments/{id}/children/ — дети конкретного родителя
GET  /api/specialist/assignments/{id}/reports/  — все отчёты родителя

# Отчёты (read-only для специалиста)
GET  /api/specialist/reports/{report_id}/      — просмотр отчёта
POST /api/specialist/reports/{report_id}/validate/  — создать ValidationRecord
GET  /api/specialist/reports/{report_id}/validate/  — посмотреть свою валидацию

# Календарная валидация
GET  /api/validation/specialist/appointments/       — консультации специалиста
GET  /api/validation/specialist/appointments/{id}/report/ — отчёт для консультации
POST /api/validation/specialist/appointments/{id}/validate/ — доменная валидация после созвона
GET  /api/validation/specialist/slots/              — свободные/забронированные слоты специалиста
POST /api/validation/specialist/slots/              — добавить слот
DELETE /api/validation/specialist/slots/{id}/       — удалить свободный слот

# Аналитика (Phase 3)
GET  /api/specialist/analytics/               — агрегированные данные

    4.1. Workflow календарной валидации¶

    1. Родитель завершает скрининг.
    2. Backend генерирует Report с доменными баллами, расшифровкой, рекомендациями и PDF-отчётом с водяным знаком.
    3. auto_assign_report(report) создаёт SpecialistAssignment и ValidationAppointment для доступного верифицированного специалиста. Сначала ищется специалист нужной роли; если такого нет, отчёт назначается любому доступному верифицированному специалисту, чтобы не зависать без проверки.
    4. Специалист получает Telegram-уведомление о новом отчёте и видит его в /specialist/queue и /specialist/calendar.
    5. Специалист добавляет свободный слот в /specialist/calendar.
    6. Родитель получает Telegram-уведомление, что нужно выбрать удобное время в отчёте.
    7. Родитель открывает /dashboard/results/{report_id} и выбирает слот в блоке консультаций.
    8. После выбора слота ValidationAppointment.status = scheduled, SpecialistAssignment.status = in_review, слот становится недоступным, а Telegram-уведомление уходит родителю и специалисту.
    9. После созвона специалист открывает /specialist/appointments/{appointment_id}/validate, подтверждает/корректирует домены и завершает валидацию.
    10. Когда все консультации по отчёту завершены, Report.validation_status = fully_validated.

    6. Верификация специалистов (Admin)¶

    Процесс верификации специалиста: 1. Специалист заполняет профиль и загружает документы (→ статус: pending) 2. Admin видит в Django Admin список pending специалистов 3. Admin проверяет документы вручную 4. Admin устанавливает verification_status = 'verified' или 'rejected' 5. Специалист получает уведомление в Telegram

    Django Admin для верификации¶

    @admin.register(SpecialistProfile)
class SpecialistProfileAdmin(admin.ModelAdmin):
    list_display = ['user', 'specialization', 'verification_status', 'verified_at']
    list_filter = ['verification_status', 'specialization']
    actions = ['verify_selected', 'reject_selected']

    def verify_selected(self, request, queryset):
        queryset.update(verification_status='verified', verified_by=request.user)

    Система собирает данные для научного исследования:

    Это позволяет сравнить: алгоритм vs. специалист → калибровка зон.

    8. Важные ограничения¶

    • Специалист видит только данные назначенных родителей (Permissions + QuerySet filter)
    • Верифицированный специалист должен иметь verification_status='verified' для валидации
    • Платформа экспериментальная — дисклеймер показывается в кабинете специалиста тоже
    • Комментарий специалиста — не медицинский диагноз, отображается как "Профессиональное наблюдение"
