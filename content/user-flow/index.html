    User Flow — Berkana Brain¶

    Обзор¶

    Каждый пользователь проходит линейный путь с guard-условиями. Пропустить этап нельзя — система перенаправляет.

    1. Полная карта пути (Parent)¶

    [Лендинг]
    │
    ▼ нажал "Начать скрининг" или "Войти"
[Страница входа — /login]
    │  Шаг 1: вводит номер телефона
    │  → POST /api/auth/request-phone/ → возвращает deep_link
    │
    ▼
[Шаг 2: открыть Telegram Bot]
    │  Бот присылает 4-значный код
    │
    ▼
[Шаг 3: ввести код на сайте]
    │  → POST /api/auth/verify-phone/ → JWT tokens
    │
    ▼ JWT получен, сессия активна
[Guard: profile_complete?]
    │ НЕТ ──────────────────────────────────────────────────────┐
    │ ДА                                                        │
    ▼                                                           ▼
[Guard: есть хотя бы 1 ребёнок?]                  [/dashboard/profile — заполни профиль]
    │ НЕТ ──────────────────────────────────────────────────────┐
    │ ДА                                                        │
    ▼                                                           ▼
[/dashboard — дашборд]                            [/dashboard/children/add — добавь ребёнка]
    │                                                           │
    │ нажал "Начать скрининг"                                   │ (после добавления)
    ▼                                                           │
[/screening/start — выбор ребёнка]   ◄──────────────────────────┘
    │  Показывает список детей с возрастом и статусом скрининга
    │  Если у ребёнка есть незавершённая сессия → "Продолжить"
    │  Иначе → "Начать скрининг"
    │
    ▼ выбрал ребёнка
[Авто-выбор модуля по age_months]
    │  GET /api/modules/?child_age={months}
    │  Возвращает подходящий ScreeningModule
    │
    ▼
[/screening/{session_id} — прохождение теста]
    │  Вопросы по одному (прогресс-бар)
    │  "Save and exit" → сохраняет сессию, редирект на дашборд
    │  Resume при возврате: продолжает с последнего отвеченного вопроса
    │
    ▼ ответил на все вопросы → нажал "Завершить"
[POST /api/sessions/{id}/complete/]
    │  Сервер: вычисляет domain_scores, интерпретации, генерирует Report
    │  Создаёт Notification(type='report_ready')
    │
    ▼
[/screening/{session_id}/feedback — опрос обратной связи]
    │  3-5 вопросов о понятности теста
    │  Не обязательный (можно пропустить), но система просит пройти
    │
    ▼
[/reports/{report_id} — страница результатов]
    │  Зоны по доменам (🟢🟡🔴)
    │  Ключевые маркеры развития
    │  Рекомендации (по зонам)
    │  Дисклеймер (экспериментальная платформа)
    │  Кнопка "Скачать PDF"
    │  Если отчёт назначен специалисту: блок "Консультации со специалистами"
    │  Родитель выбирает слот после того, как специалист добавит свободное время
    │
    ▼ нажал "Скачать PDF"
[GET /api/reports/{id}/pdf/]
    │  Отдаёт PDF с водяным знаком
    │
    ▼
[/dashboard — вернуться в кабинет]

    2. Guard-условия (ProtectedRoute логика)¶

    Route Guard Redirect
    /dashboard/* isAuthenticated /login
    /screening/* profile_complete AND children.count > 0 → профиль или → добавить ребёнка
    /reports/* isAuthenticated /login
    /specialist/* role === 'specialist' /dashboard

    Реализация в ProtectedRoute.tsx: 

    // Текущий guard (только auth):
if (!isAuthenticated) return <Navigate to="/login" />;

// Нужно добавить:
if (requireProfile && !profile?.profile_complete) return <Navigate to="/dashboard/profile" />;
if (requireChild && childrenCount === 0) return <Navigate to="/dashboard/children/add" />;
if (requireRole && profile?.role !== requireRole) return <Navigate to="/dashboard" />;

    3. Путь специалиста¶

    [Лендинг] → [/login] → JWT (role='specialist')
    │
    ▼
[/specialist/dashboard — список клиентов]
    │  Таблица: Клиент | Ребёнок | Дата скрининга | Статус валидации
    │
    ▼ нажал на клиента
[/specialist/clients/{parent_id}/children]
    │  Список детей и их скринингов
    │
    ▼ нажал на отчёт
[/specialist/reports/{report_id} — просмотр отчёта]
    │  Read-only просмотр отчёта с зонами
    │  Форма валидации: подтвердить / опровергнуть / добавить комментарий
    │
    ▼ заполнил форму
[POST /api/specialist/reports/{id}/validate/]
    │  Создаёт ValidationRecord
    │  Уведомляет родителя через Telegram
    │
    ▼
[/specialist/profile — профиль специалиста]
    │  Загрузка документов (лицензия, диплом, сертификат)
    │  Статус верификации: Не верифицирован / Pending / Verified

    3.1. Календарная валидация отчёта¶

    [Родитель завершил скрининг]
    │
    ▼
[Report создан + PDF с водяным знаком доступен родителю]
    │
    ▼
[auto_assign_report]
    │  Создаёт SpecialistAssignment + ValidationAppointment
    │  Отчёт появляется в кабинете специалиста
    │  Специалист получает Telegram-уведомление
    ▼
[/specialist/calendar]
    │  Специалист добавляет свободный слот
    │
    ▼
[Telegram родителю]
    │  "Специалист добавил свободное время, выберите слот"
    │
    ▼
[/dashboard/results/{report_id}]
    │  Родитель выбирает слот в блоке консультаций
    │
    ▼
[ValidationAppointment.status = scheduled]
    │  Telegram родителю + специалисту:
    │  "Созвон по валидации назначен"
    │
    ▼
[/specialist/appointments/{appointment_id}/validate]
    │  Специалист проводит валидацию после созвона
    ▼
[Report.validation_status = fully_validated]

    4. Уведомления и follow-up¶

    Событие Получатель Статус
    Авторизация Родитель/специалист/admin ✅ Telegram code/deep link
    Новый специалист подал документы Admin ✅ Telegram admin notification
    Специалист верифицирован/отклонён Специалист ✅ Telegram result notification
    Новый отчёт назначен Специалист ✅ Telegram assignment notification
    Специалист добавил слот Родитель ✅ Telegram slot notification
    Слот выбран Родитель + специалист ✅ Telegram scheduled notification
    Повторный скрининг Родитель 🟡 next_rescreen_at считается; admin видит родителей через screening=rescreen_due; авто-уведомление родителю ещё не реализовано

    5. Навигация внутри дашборда (родитель)¶

    /dashboard                          — Главная (статистика + последние скрининги)
/dashboard/profile                  — Профиль пользователя
/dashboard/children                 — Список детей
/dashboard/children/add             — Добавить ребёнка
/dashboard/screenings               — История скринингов
/dashboard/results                  — Список отчётов
/dashboard/results/{id}             — Детальный отчёт ← ЗАМЕНЯЕТ /reports/{id}
/dashboard/notifications            — Уведомления
/dashboard/history                  — История действий

    6. Правило "экспериментальной платформы"¶

    На каждом из следующих экранов виден дисклеймер: - Страница входа (в footer) - Страница скрининга (в header опросника) - Страница результатов (перед началом отчёта) - PDF-отчёт (на каждой странице) - Лендинг (в Legal секции)

    Текст: "Berkana Brain — экспериментальная платформа для сбора данных и апробации методики нейрофункционального скрининга. Результаты не являются медицинским диагнозом и не заменяют консультацию специалиста."
