    Компоненты¶

    1. React SPA (Frontend)¶

    • SPA (Single Page Application) — весь роутинг на клиенте
    • Общается с backend через axios (REST API)
    • JWT-токены хранятся в localStorage
    • Auto-refresh токенов через axios interceptor
    • Состояние: AuthContext (session) + TanStack Query (серверные данные)

    2. Django REST API (Backend)¶

    • Stateless REST API, отдаёт JSON
    • JWT authentication (simplejwt): access token (15 мин) + refresh token (7 дней)
    • Django Admin panel (/admin/) для управления контентом модулей
    • Бизнес-логика: алгоритм скоринга, генерация отчётов, планирование уведомлений

    3. Telegram Bot¶

    • Работает через webhook (продакшн) или polling (разработка)
    • Две задачи:
    • Отправка кода авторизации при регистрации/входе
    • Уведомления о повторном скрининге (Фаза 2)
    • Не хранит состояние — всё хранится в Django БД

    4. PostgreSQL¶

    • Единственное хранилище данных
    • Permissions на уровне Django (не RLS) — каждый view проверяет права
    • JSONField для мультиязычных текстов вопросов/интерпретаций

    Поток данных: Скрининг¶

    1. Родитель → выбирает ребёнка на `/dashboard/screenings`
2. Frontend → POST /api/screening/sessions/ { child }
3. Django → находит активный ScreeningModule по возрасту ребёнка
4. Django → создаёт или возвращает ScreeningSession (status='in_progress')
5. Frontend → открывает `/questionnaire?session={id}`
6. Frontend → GET /api/screening/sessions/{id}/questions/
7. Родитель → отвечает на вопросы
8. Frontend → POST /api/screening/sessions/{id}/answers/ (upsert)
9. Родитель → завершает тест
10. Frontend → POST /api/screening/sessions/{id}/complete/
11. Django → generate_report() → создаёт базовый Report
12. Frontend → `/questionnaire/result?session={id}&report={id}` (success stub до Недели 6)

    Поток данных: Авторизация¶

    1. Пользователь → вводит телефон на /login
2. Frontend → POST /api/auth/request-phone/ { phone }
3. Django → создаёт PhoneAuthSession и возвращает { session_token, deep_link }
4. Frontend → сохраняет auth-сессию в sessionStorage и открывает Telegram deep link
5. Telegram Bot → /start TOKEN, привязывает telegram_id к PhoneAuthSession
6. Пользователь → делится контактом в Telegram
7. Bot → проверяет phone == session.phone, генерирует код, кладёт в Django cache
8. Пользователь → вводит код на сайте
9. Frontend → POST /api/auth/verify-phone/ { session_token, phone, code }
10. Django → проверяет код, переиспользует/создаёт User+UserProfile
11. Django → возвращает { access_token, refresh_token }
12. Frontend → сохраняет JWT в localStorage, редирект на /dashboard

    Безопасность¶

    • HTTPS обязателен в продакшне (Telegram webhook требует HTTPS)
    • CORS настроен для dev frontend origins из backend/.env
    • JWT с коротким TTL access token (15 мин)
    • PhoneAuthSession + Django cache — одноразовый phone auth flow, TTL 10 мин, код не хранится в БД
    • Все API-endpoints требуют аутентификации (кроме auth endpoints)
    • Django permissions: пользователь видит только свои данные
